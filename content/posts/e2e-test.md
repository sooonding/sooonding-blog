## E2E 테스트, 왜 필요할까?


친구가 "유닛 테스트 열심히 짜고 있는데, E2E 테스트까지 해야 해?"라고 물어봤다고 생각해보자.


### 먼저 유닛 테스트가 뭔지부터


유닛(Unit) 테스트는 **가장 작은 단위의 코드**를 테스트하는 거다. 함수 하나, 컴포넌트 하나가 의도대로 동작하는지 확인한다.


```javascript
// 이메일 검증 함수
function isValidEmail(email) {
  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
}

// 유닛 테스트
test('올바른 이메일 형식을 검증한다', () => {
  expect(isValidEmail('test@example.com')).toBe(true);
  expect(isValidEmail('invalid')).toBe(false);
});
```


이렇게 작은 조각들이 제대로 작동하는지 빠르게 검증한다. Jest, Vitest 같은 도구로 작성한다.


### 그럼 E2E 테스트가 뭔데?


E2E(End-to-End) 테스트는 말 그대로 **처음부터 끝까지** 테스트하는 거다. 사용자가 브라우저 열고, 버튼 누르고, 페이지 이동하고, 데이터 입력하는 그 모든 과정을 실제로 시뮬레이션한다.


**Cypress나 Playwright** 같은 도구가 브라우저를 띄워서 "클릭 → 입력 → 이동 → 확인"을 자동으로 수행한다. 사람이 직접 클릭하는 걸 코드로 대신하는 셈이다.


### 레고로 비유하면


유닛 테스트는 레고 블록 하나하나가 멀쩡한지 확인하는 거다. "이 블록 색깔 맞아?", "끼우는 부분 안 부러졌어?" 이런 검사.


E2E 테스트는 그 블록들로 **집을 완성한 뒤** 문이 열리는지, 지붕이 안 무너지는지 확인하는 거다.


블록이 다 멀쩡해도 조립하면 문제가 생길 수 있다. 문틀이랑 문 크기가 안 맞는다든지.


### 실제 예시: 로그인 기능


유닛 테스트가 확인하는 것:

- 이메일 형식 검증 함수가 제대로 동작하는가
- 비밀번호 암호화 로직이 맞는가

E2E 테스트가 확인하는 것:

- 사용자가 이메일 입력 → 비밀번호 입력 → 로그인 버튼 클릭 → 대시보드로 이동 → 세션 유지

유닛 테스트에서 각 함수가 다 통과해도, 실제로 로그인 버튼 누르면 페이지가 안 넘어갈 수 있다. 라우팅 설정 문제, API 응답 처리 누락, 상태 관리 버그 같은 게 숨어있을 수 있으니까.


### 언제 E2E 테스트를 쓰면 좋을까?


모든 페이지에 E2E 테스트를 붙일 필요는 없다. **사용자에게 직접 영향을 주는 핵심 흐름**에 집중하면 된다.

- 회원가입 → 로그인 → 메인 페이지 진입
- 상품 선택 → 장바구니 → 결제 완료
- 폼 작성 → 제출 → 결과 확인

이런 **크리티컬 패스**가 깨지면 서비스 자체가 안 돌아가니까, 여기에 E2E 테스트를 걸어두면 배포 전에 치명적인 문제를 잡을 수 있다.


### 그래서 둘 다 필요하다


| 유닛 테스트      | E2E 테스트         |
| ----------- | --------------- |
| 빠르다         | 상대적으로 느리다       |
| 디버깅 쉽다      | 어디서 터졌는지 찾기 어렵다 |
| 개별 로직 검증    | 전체 흐름 검증        |
| 많이 짜도 부담 적다 | 핵심 시나리오 위주로     |


유닛 테스트로 촘촘하게 깔고, E2E 테스트로 핵심 사용자 여정을 잡는 게 현실적인 전략이다.


### 한 줄 정리


유닛 테스트는 부품 검수, E2E 테스트는 완성품 검수. 둘 다 해야 출고 전에 불량 잡는다고 생각
